三、Step 2 — BatchRunStore SQLite 遷移（Commit: 571e644）
---------------------------------------------------------
完整改寫 src/core/batch_run_store.py，從 JSON 檔案遷移至 SQLite。

【Schema 設計】
  三張資料表：
    batch_runs    — 批次 meta（type, parent_run_id, dataset_label, recipe_ids...）
    image_results — 每張圖一行（image_path, image_id, status, overlay_path...）
    measurements  — 每筆量測一行（calibrated_nm, feature_type, state_name...）
  六個索引：batch_id, recipe_id, calibrated_nm, start_time, parent_run_id 等。

【關鍵實作】
  _get_conn()
    - threading.local() 儲存 thread-local connection
    - WAL journal mode：允許 QThread 並發讀寫不互鎖
    - synchronous=NORMAL：效能與安全的平衡
    - foreign_keys=ON：cascade delete 保護

  save(batch_run) → str
    - 單一 with conn: transaction
    - executemany() 批量寫入 image_results & measurements（10-100x 加速）
    - 回傳 batch_id（str），原版回傳 Path

  save_multi(mbr) → "multi_{run_id}"
    - 同一 transaction 寫入 parent row + N 個 child dataset rows
    - child rows 的 parent_run_id = parent batch_id

  load(batch_id)
    - 依 type 欄位分派 _load_single() 或 _load_multi()

  list_runs()
    - WHERE parent_run_id IS NULL：只顯示 top-level batch
    - "file_path" 欄位存 batch_id（向後相容現有呼叫方）

  delete(batch_id)
    - multi type：先刪所有 child rows（measurements/image_results/batch_runs）
    - 再刪 parent row；ON DELETE CASCADE 為備援

  get_stats_for_recipe()
    - SQL JOIN batch_runs ⋈ measurements
    - Python statistics.stdev()（SQLite 無內建 STDEV）

【API 相容性】
  batch_workspace.py、report_workspace.py、klarf_export_dialog.py 無需改動：
  這三個檔案都透過 summaries[row]["file_path"] 呼叫 store.load()，
  現在 "file_path" 存的是 batch_id，load() 也改吃 batch_id，透明相容。

【測試更新】
  tests/test_batch_run_store.py
    - BatchRunStore(runs_dir=tmp_path) → BatchRunStore(db_path=tmp_path/"test.db")
    - save() 回傳 str batch_id（移除 path.exists() 斷言）
    - load/delete 改用 batch_id 字串

  tests/test_history.py
    - 同上介面更新

五、Bug Fix Round3 — SQLite 遷移後回歸修復（Commit: 5e4b399）
--------------------------------------------------------------
遷移完成後發現兩個功能回歸：

  Bug 1 | state_name / structure_name load 後被清空
    根因：_load_single() 重建 measurement dict 時，state_name/structure_name
          硬寫為 ""，因為 measurements table 原本沒有這兩個欄位。
    影響：ResultsPanel 狀態篩選器預設值崩潰；export 時 MeasurementRecord.state_name
          全為空，破壞與舊版 JSON store 的 round-trip 等效性。
    修復：
      1. measurements table 新增：
           state_name     TEXT DEFAULT ''
           structure_name TEXT DEFAULT ''
      2. _INSERT_MEAS SQL + _meas_tuple() 加入兩欄
      3. _load_single() 改從 DB 欄位讀取
      4. _init_db() 加 ALTER TABLE ADD COLUMN（try/except，對舊 DB 向後相容）

  Bug 2 | multi-dataset 子 run 被排除在歷史統計之外
    根因：get_stats_for_recipe() 的 WHERE 條件為 parent_run_id IS NULL，
          但 multi run 的 measurements 存在子 batch rows（parent_run_id='multi_*'），
          全部被排除。History Run Chart / 統計摘要在只有 multi-dataset 歷史時全空白。
    修復：將 WHERE parent_run_id IS NULL 改為 WHERE type != 'multi'
          - 獨立 batch（type='single'）照舊納入
          - multi 子 dataset（type='single', parent_run_id='multi_*'）現正確納入
          - multi parent row（type='multi', 無 measurements）透過 JOIN 自然排除

  新增迴歸測試（tests/test_history.py::TestRegressionFixes）：
    test_state_name_round_trips          — state_name/structure_name save→load 等效
    test_state_name_migration_defaults   — 舊 DB 缺欄位時預設空字串
    test_multi_run_included_in_stats     — multi 子 dataset 出現在 stats 結果中

八、重要設計決策摘要
--------------------

     確保現有 DB 不需手動 schema upgrade。
